SECTION 1 — FRONTEND FRAMEWORK UPDATE
Verdict: ✅ CONFIRMED — Next.js 15 with targeted upgrades
In 2026, Next.js (now at version 15+) remains the gold standard for building fast, SEO-friendly web applications. With advancements in Turbopack, which optimizes build speeds, and enhanced Server Actions for secure, efficient server-side operations in Next.js 15, it has solidified its position as an efficient tool for web development. NSP
Next.js is the most widely adopted React framework in production as of 2026. Next.js applications typically achieve 40–60% faster Largest Contentful Paint (LCP) scores than equivalent React SPAs because content is pre-rendered server-side and images are automatically optimized. The gap widens on mobile devices and slower connections. Education for All in India
For ScholarLogic specifically — where 60% of users are on mobile in Tier 2/3 Indian cities — this LCP gap is not cosmetic. It is the difference between a student waiting 4 seconds and a student waiting 1.8 seconds on a 4G connection. Next.js stays.
2026 Best Practice Upgrades to Apply:
Partial Prerendering (PPR) — New in Next.js 15 PPR is the single most impactful Next.js upgrade for ScholarLogic's architecture. It allows a single page to have a static shell (served instantly from CDN) with async streaming slots for dynamic content (match list, stats cards).
typescript
// app/dashboard/page.tsx — Enable PPR
import { unstable_noStore as noStore } from 'next/cache'
import { Suspense } from 'react'
// Static shell renders at build time (CDN-cached)
export default function DashboardPage() {
  return (
    <DashboardShell>
      {/* Dynamic slots stream in as they resolve */}
      <Suspense fallback={<StatCardsSkeleton />}>
        <StatCards />        {/* Fetches from DB, streams in */}
      </Suspense>
      <Suspense fallback={<TopMatchSkeleton />}>
        <TopMatch />         {/* Fetches AI match score, streams in */}
      </Suspense>
    </DashboardShell>
  )
}
Result: Dashboard shell visible in ~200ms, dynamic data streams in as resolved. No full-page loading state.
Server Actions (Replace Route Handlers for Mutations) Server Actions now have unguessable endpoints and automatic removal of unused actions in Next.js 15. NSP Replace all form-based Route Handler mutations (profile save, application status update, document upload trigger) with Server Actions — eliminating the client fetch layer entirely.
typescript
// Before (v1.0): Client → fetch('/api/user/profile') → Route Handler
// After (2026):  Client → Server Action (co-located, type-safe, no API call)
'use server'
export async function updateProfile(formData: FormData) {
  const { userId } = await getAuth()
  const validated = profileSchema.parse(Object.fromEntries(formData))
  await db.user.update({ where: { clerkId: userId }, data: validated })
  revalidatePath('/dashboard/profile')
  // Triggers BullMQ match recalculation job
  await matchQueue.add('recalculate', { userId, changedFields: Object.keys(validated) })
}

Add Trigger.dev v3 for Background Jobs (Replaces Raw BullMQ)
If you plan on running long-running workloads such as data processing, ETL jobs, large file imports/exports, user operations like account deletion, data merging, and stat recalculations — deploying backend services to Vercel functions will not be the right fit. Taction Soft
BullMQ is robust but requires self-managing a Redis connection, worker processes, and a separate monitoring UI. Trigger.dev v3 (2025) is a purpose-built background job platform that runs in the same Next.js codebase with zero infrastructure setup:
typescript
// Before (BullMQ) — requires Redis server + worker process + Bull Board
import { Queue, Worker } from 'bullmq'
const matchQueue = new Queue('match-recalculation', { connection: redis })
// Separate worker.ts process to deploy and monitor...
// After (Trigger.dev v3) — co-located, auto-deploys, has built-in UI
import { task } from "@trigger.dev/sdk/v3"
export const recalculateMatches = task({
  id: "recalculate-matches",
  retry: { maxAttempts: 3, backoffFactor: 2 },
  run: async ({ userId, changedFields }: { userId: string; changedFields: string[] }) => {
    // Same matching logic — no infrastructure change
    const matches = await runMatchingEngine(userId, changedFields)
    await cacheMatchResults(userId, matches)
  }
})
// Trigger from Server Action:
await recalculateMatches.trigger({ userId, changedFields })
Trigger.dev provides: built-in retry logic, real-time job monitoring UI, replay failed jobs, job scheduling (replaces APScheduler in Python pipeline), and serverless execution that scales to zero. Cost: $0 for first 50,000 runs/month — covers ScholarLogic through Month 6.
Keep BullMQ only for the Redis pub/sub log streaming feature (pipeline live logs) where Trigger.dev is overkill.

## SECTION 4 — DATABASE CHOICE UPDATE
### Verdict: ⚠️ MAJOR RECONSIDERATION — Evaluate Supabase (PostgreSQL) vs MongoDB Atlas
This is the most significant change from v1.0. The database landscape shifted substantially in 2025.
**The 2026 PostgreSQL Reality Check**
PostgreSQL is the safer default for most full-stack apps in 2026. PostgreSQL shows broader adoption — 55.6% of developers in the 2025 Stack Overflow data — and strong job growth, about a 73% increase in postings with roughly a 12% pay premium. 
PostgreSQL's JSONB support has made the "I need flexible JSON" argument for MongoDB much weaker — PostgreSQL handles JSON extremely well while also giving you joins, transactions, and SQL. The PostgreSQL ecosystem (Supabase, Neon, pgvector) has exploded, making it excellent for both traditional apps and AI-era applications. 
**The Honest MongoDB Defense for ScholarLogic**
MongoDB dominates when the data model evolves quickly or when performance depends on horizontal scaling. In 2025, MongoDB has evolved into a data platform rather than a simple database — supporting Atlas Search, Vector Search, and time-series ingestion for AI-ready workloads. 
The one caveat: if your team is already deeply experienced in MongoDB and you're building something genuinely document-oriented — a product catalog with wildly variable attributes, a logging system — MongoDB is absolutely the right choice. 
**Decision Framework for ScholarLogic:**
| Factor | MongoDB Wins | PostgreSQL Wins | ScholarLogic Verdict |
|---|---|---|---|
| Scholarship variable schema | ✅ Document model natural | ⚠️ JSONB workaround | MongoDB wins |
| User.documents embedded array | ✅ Native | ⚠️ Separate table needed | MongoDB wins |
| Application clerkId+scholarshipId unique | ⚠️ Manual index | ✅ Foreign key + constraint | PostgreSQL wins |
| Complex eligibility analytics | ⚠️ Aggregation verbose | ✅ SQL window functions | PostgreSQL wins |
| pgvector for semantic search | ❌ Atlas Vector ($$) | ✅ Free extension | PostgreSQL wins |
| Row-level security for multi-tenant | ❌ Not native | ✅ Built-in RLS | PostgreSQL wins |
| Hiring pool | ⚠️ Smaller | ✅ Broader | PostgreSQL wins |
| Mumbai region latency | ✅ Atlas ap-south-1 | ✅ Supabase ap-south-1 | Tie |
**Recommended Path: Hybrid Architecture**
Rather than choosing one or the other, the 2026 best practice for ScholarLogic's specific data profile is a hybrid:
```
PRIMARY: Supabase (PostgreSQL) — for structured relational data
  ├── users table (with JSONB documents column)
  ├── applications table (with proper foreign keys + unique constraint)
  ├── match_scores table
  └── processed_pdfs table
SECONDARY: MongoDB Atlas — for semi-structured extraction data only
  └── scholarships collection (variable schema, DeepSeek output, document model)
This gives PostgreSQL's ACID guarantees, RLS, pgvector, and SQL power for everything that benefits from relational structure, while MongoDB handles the one genuinely document-oriented collection (scholarships with wildly variable eligibility structures).
If choosing one only: Supabase (PostgreSQL)
The best database for a startup building web-based SaaS in 2026 is Supabase. It combines PostgreSQL with built-in authentication, Row Level Security for multi-tenancy, real-time subscriptions, file storage, and edge functions. You get a complete backend without stitching together five services. The free tier is generous enough to validate your idea, and the pricing scales predictably. Fastweb
Supabase in Mumbai (ap-southeast-1 or ap-south-1) provides:
PostgreSQL with pgvector (semantic scholarship search in Phase 3 — no Pinecone needed)
Built-in auth (can complement or partially replace Clerk for non-SSO users)
Realtime subscriptions (replaces SSE for pipeline log streaming in a simpler model)
Row Level Security for student data isolation
Supabase Storage as a Cloudinary complement for smaller files
ORM Migration: Mongoose → Prisma 6
Regardless of database choice, replace Mongoose with Prisma 6. Prisma ORM generates TypeScript types from your database schema and provides a query builder. It connects to any Postgres database (Supabase, Neon, PlanetScale, etc.). Fastweb
Prisma works with both PostgreSQL (Supabase) AND MongoDB — enabling a smooth transition if the team starts with MongoDB and migrates later:
typescript
// prisma/schema.prisma — works with both MongoDB and PostgreSQL
generator client {
  provider = "prisma-client-js"
}
datasource db {
  provider = "postgresql"  // Switch to "mongodb" if needed — same schema
  url      = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")  // Required for Supabase connection pooling
}
model User {
  id          String   @id @default(cuid())
  clerkId     String   @unique
  email       String   @unique
  documents   Json[]   // JSONB array in PostgreSQL — identical to MongoDB embedded
  profileCompletion Int @default(0)
  createdAt   DateTime @default(now())
  applications Application[]
  matchScores  MatchScore[]
}
model Application {
  id            String @id @default(cuid())
  clerkId       String
  scholarshipId String
  status        ApplicationStatus @default(SAVED)
  savedAt       DateTime @default(now())
  appliedAt     DateTime?
  wonAt         DateTime?
  user          User @relation(fields: [clerkId], references: [clerkId])
  @@unique([clerkId, scholarshipId])  // Enforced at DB level — same as MongoDB unique index
  @@index([clerkId, status])
}
enum ApplicationStatus {
  SAVED
  APPLIED
  UNDER_REVIEW
  WON
  REJECTED
}
Benefits of Prisma over Mongoose:
Generated TypeScript types from schema — zero runtime type mismatches
Type-safe query builder with auto-complete — faster development
Migration system (prisma migrate) — trackable, reversible schema changes
Works with PostgreSQL AND MongoDB — no rewrite if database changes
Prisma Studio — visual database browser built-in

import { google } from '@ai-sdk/google'
import { anthropic } from '@ai-sdk/anthropic'
export async function generateSOP(prompt: string) {
  try {
    return await streamText({
      model: google('gemini-2.5-flash'),
      prompt
    })

This is the most cost-impactful update. The deployment landscape shifted dramatically in 2025–2026.
**The Vercel Tax — Real Numbers**
One real migration showed: Vercel Bill: $850/month vs Coolify + Hetzner Bill: $14/month. This is what is called the "Vercel Tax" — the premium you pay for not thinking about servers. But when you scale, that abstraction tax becomes heavier than the infrastructure cost. 
Railway typically costs less than Vercel, particularly for applications with consistent traffic patterns. One documented case showed a startup facing a $2,000 monthly bandwidth bill on Vercel that was impacting their margins. Railway's pay-per-use model can save approximately 40% compared to other cloud providers. 
**Three-Phase Deployment Strategy:**
```
PHASE 1 (Month 1-4, <2,000 users): Vercel Free/Pro
├── Benefit: Zero-friction deployment, instant preview URLs
├── Cost: $0-20/month
└── Trigger to move: When monthly Vercel bill approaches $100
PHASE 2 (Month 5-9, 2,000-20,000 users): Vercel Pro + Railway Full
├── Move Python pipeline, BullMQ, long-running jobs to Railway
├── Keep Next.js on Vercel (still best DX for Next.js)
├── Cost: $30-80/month total
└── Trigger to move: When Vercel bill approaches $300/month
PHASE 3 (Month 10+, 20,000+ users): Railway Full Stack
├── Migrate Next.js FROM Vercel TO Railway
├── Railway handles Next.js via long-running container (no cold starts)
├── Cost: $40-120/month (vs $300-500+ on Vercel)
└── Benefit: Persistent connections, no 10s function timeout, WebSocket support
```
**Why Railway beats Vercel for Phase 3:**
Railway uses a custom builder that takes your source code or Dockerfile and automatically builds and deploys it, without configuration. Your code runs on a long-running server, making it ideal for apps that need to stay running or maintain a persistent connection. All deployments come with smart defaults out of the box. 
ScholarLogic specifically needs long-running servers for: SSE log streaming (pipeline monitor), BullMQ worker processes, and Python pipeline scheduler. Deploying backend services requiring persistent connections to Vercel functions will not be the right fit. 
**Ultimate Phase 4 — Self-Hosted Coolify on Hetzner (Optional, Month 18+):**
Stick with Vercel if you have under 1,000 users and are pre-revenue. Switch to Coolify if you have found Product-Market Fit, are burning cash on bandwidth, and want data sovereignty. 
Hetzner CX32 (€10.52/month, 4 vCPU, 8GB RAM) running Coolify can host the entire ScholarLogic stack — Next.js, Python pipeline, Redis, and all worker processes — for under €20/month total. Viable at Month 18+ when the team has DevOps capacity.

### 7.1 — ADD: Turborepo (Monorepo Management)
As ScholarLogic grows to include a mobile app, admin panel, and shared utilities, managing them as separate repositories creates duplication and versioning hell. Turborepo organizes everything into a single monorepo with intelligent build caching:
```
scholarlogic/
├── apps/
│   ├── web/          ← Next.js (current repo becomes this)
│   ├── mobile/       ← React Native (Month 9+)
│   └── admin/        ← Dedicated admin panel (Month 12+)
├── packages/
│   ├── ui/           ← Shared shadcn/ui components
│   ├── db/           ← Prisma schema + client (shared between web + mobile API)
│   ├── ai/           ← Shared AI utilities (SOP prompts, matching logic)
│   └── types/        ← Shared TypeScript types
└── turbo.json
json
// turbo.json
{
  "pipeline": {
    "build": { "dependsOn": ["^build"], "outputs": [".next/**", "dist/**"] },
    "dev":   { "cache": false, "persistent": true },
    "test":  { "dependsOn": ["^build"] },
    "lint":  {}
  }
}
Turborepo build cache means: when only the mobile app changes, the web app's Next.js build is served from cache — CI/CD goes from 8 minutes to 45 seconds.
